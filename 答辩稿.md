# MinimalHealth Pro 答辩稿

> 时长：5-8分钟 | 建议配合 App 演示操作

---

## 一、开场（30秒）

各位老师好，我做的项目叫 **MinimalHealth Pro——极简健康管家**。

这是一款面向日常健康管理的 Android App，核心功能包括运动打卡、饮食记录、饮水追踪、健康数据分析，以及智能定时提醒。项目的技术路线是 **Vue3 + SpringBoot + Capacitor 混合架构**，前端代码通过 Capacitor 打包进 Android WebView，同时充分利用 Android 原生能力实现了四大组件。

---

## 二、技术架构（1分钟）

整个系统分为三层：

```
┌─────────────────────────────────┐
│  Vue3 前端（17个页面）            │
│  TypeScript + Vite 构建          │
├─────────────────────────────────┤
│  Capacitor Bridge               │
│  JS ↔ Java 双向通信              │
├──────────────┬──────────────────┤
│ Android 原生  │  SpringBoot 后端  │
│ 四大组件      │  REST API        │
│ SQLite 本地   │  H2 数据库       │
└──────────────┴──────────────────┘
```

- **前端**：Vue3 + TypeScript，17个视图页面，包括仪表盘、运动列表、沉浸式跑步、饮食记录、饮水、健康分析、时间线、个人中心等
- **桥接层**：Capacitor 7，通过 Plugin 机制实现前端 JS 调用 Android 原生能力
- **Android 原生**：完整实现了四大组件——Activity、Service、BroadcastReceiver、ContentProvider
- **后端**：SpringBoot 3，25个 REST API，JWT 鉴权，10张数据表

---

## 三、四大组件详解（核心，3分钟）

### 3.1 Activity（活动）— 2个

**MainActivity.java**（容器 Activity）
- 继承 Capacitor 的 BridgeActivity，作为整个 App 的入口
- 加载 WebView 渲染 Vue3 前端页面
- 在 `onCreate` 中注册 NativeBridge 插件，打通 JS 到 Java 的调用通道

**AboutActivity.java**（原生 Activity）— **演示点**
- 一个纯原生 Android Activity，使用 XML 布局（activity_about.xml）
- 展示"健康中国2030"思政元素，包含习近平总书记语录：
  > "没有全民健康，就没有全面小康"
- 展示项目简介和技术栈信息
- 入口在 App 个人页底部"关于我们"按钮

**演示操作**：打开 App → 个人页 → 点击"关于我们"→ 跳转到原生 AboutActivity

**调用链路**：
```
ProfileView.vue (点击按钮)
  → NativeBridge.openAbout()      // TypeScript
  → Capacitor Plugin 桥接         // 自动序列化
  → NativeBridgePlugin.java       // Java Plugin
  → startActivity(AboutActivity)  // 原生跳转
```

---

### 3.2 Service（服务）— 1个

**HealthReminderService.java**

- 继承 `android.app.Service`，在后台持续运行
- **每15分钟**检查一次当前时间，匹配提醒规则后发送系统通知
- 三类提醒：
  - 💧 **饮水提醒**：9:00-21:00之间，整点提醒喝水
  - 🏃 **运动提醒**：每天10:00和15:00提醒运动
  - 🌙 **睡眠提醒**：每天21:30提醒休息
- 返回 `START_STICKY`，被杀后系统自动重启
- 通知点击后跳回 MainActivity

**演示操作**：展示通知栏的健康提醒推送（可以让老师看到 Notification）

---

### 3.3 BroadcastReceiver（广播接收器）— 2个

**BootReceiver.java** — 开机自启
- 监听 `BOOT_COMPLETED` 广播
- 手机开机后自动启动 HealthReminderService
- 需要 `RECEIVE_BOOT_COMPLETED` 权限

**ConnectivityReceiver.java** — 网络监听
- 监听 `CONNECTIVITY_CHANGE` 广播
- 网络恢复时自动重启提醒服务
- 网络断开时记录日志

**演示操作**：重启模拟器 → App 自动启动服务（不需要手动打开 App）

---

### 3.4 ContentProvider（内容提供者）— 1个

**HealthDataProvider.java**

- 继承 `android.content.ContentProvider`
- 对外暴露 URI：`content://com.minimalhealth.app.healthprovider/stats`
- 支持完整的 CRUD 操作：`query()`、`insert()`、`update()`、`delete()`
- 底层使用 `HealthDatabaseHelper` 管理 SQLite 数据库
- 本地 SQLite 表结构（`health_stats`）：

| 字段 | 类型 | 含义 |
|------|------|------|
| _id | INTEGER PK | 主键 |
| record_date | TEXT UNIQUE | 记录日期 |
| steps | INTEGER | 步数 |
| resting_hr | INTEGER | 静息心率 |
| sleep_hours | REAL | 睡眠时长 |
| water_ml | INTEGER | 饮水量 |
| calories | INTEGER | 消耗卡路里 |
| health_score | INTEGER | 健康评分 |

- `exported="true"`，第三方 App 可通过 ContentResolver 跨进程访问

**演示操作**：可以说明"任何第三方健康 App 都可以通过标准 ContentProvider 接口读取本 App 的数据"

## 四、AndroidManifest 注册总览（30秒）

四大组件全部在 `AndroidManifest.xml` 中正确注册：

```xml
<!-- Activity -->     第13-33行    MainActivity + AboutActivity
<!-- Service -->      第37-39行    HealthReminderService
<!-- Receiver -->     第43-58行    BootReceiver + ConnectivityReceiver
<!-- Provider -->     第62-65行    HealthDataProvider
```

权限声明：
- `INTERNET` — 网络访问
- `RECEIVE_BOOT_COMPLETED` — 开机自启
- `ACCESS_NETWORK_STATE` — 网络状态监听
- `POST_NOTIFICATIONS` — 发送通知

## 五、组件间协作（30秒）

```
开机 → BOOT_COMPLETED 广播
  → BootReceiver.onReceive()
    → startService(HealthReminderService)
      → 每15分钟 checkAndNotify()
        → NotificationManager.notify() → 通知栏弹出

网络变化 → CONNECTIVITY_CHANGE 广播
  → ConnectivityReceiver.onReceive()
    → 判断网络状态 → 启动/停止服务

用户点击"关于我们" → Capacitor JS Bridge
  → NativeBridgePlugin.openAbout()
    → startActivity(AboutActivity) → 原生关于页

第三方App → ContentResolver.query(
    content://com.minimalhealth.app.healthprovider/stats
) → HealthDataProvider.query() → SQLite → 返回 Cursor
```

4. **完整的后台服务**：Service + BroadcastReceiver 实现开机自启 + 网络监听 + 定时通知，进程被杀后自动恢复

## 七、常见答辩问题准备

**Q1：你的 App 和纯 H5 网页打包的 App 有什么区别？**
> 区别在于我使用了 Android 原生四大组件。纯 H5 打包的 App 只有一个 WebView Activity，没有任何原生能力。而我的 App 有后台 Service 发通知、BroadcastReceiver 监听系统广播、ContentProvider 提供跨进程数据共享，这些都是纯 H5 做不到的原生能力。

**Q2：四大组件中哪个最难实现？**
> ContentProvider 最复杂。需要实现完整的 CRUD 四个方法，处理 URI 匹配（UriMatcher），管理 SQLite 数据库，通知 ContentResolver 数据变更。而且 exported=true 意味着要考虑安全性，不能随意暴露数据。

**Q3：Capacitor 插件注册为什么要在 super.onCreate() 之前？**
> 因为 BridgeActivity 的 onCreate 内部会调用 load() → bridgeBuilder.create() 完成 Bridge 构建。如果 registerPlugin 在 super.onCreate() 之后调用，Bridge 已经创建完毕，插件就加不进去了。这是一个 Capacitor 7 的常见坑。

**Q4：你的 Service 会不会被系统杀死？**
> 会。但我返回了 START_STICKY，系统会在资源充足后自动重建 Service。同时 BootReceiver 和 ConnectivityReceiver 提供了双重保障——开机和网络恢复时都会重新启动服务。
